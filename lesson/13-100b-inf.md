## 100+B 인퍼런스 ##

100B(1000억) 파라미터 규모의 대형 멀티모달 모델을 아마존 EC2 GPU 인스턴스에서 인퍼런스(추론) 용도로 올리려면 필요 VRAM 계산, 인스턴스선택, 추론 프레임워크 사전에 설정 및 고려해야한다. 
지식 증류(Knowledge Distillation)나 고품질 합성 데이터(Synthetic Data) 생성을 위한 모델의 경우 다음 사항을 고려하여 양자화 여부를 결정한다.

#### ① 확률 분포(Logits)의 미세한 왜곡 ####
지식 증류(Distillation)는 단순히 정답 텍스트만 넘겨주는 것이 아니라, 모델이 정답과 오답 사이에서 고민하는 Soft Target(Logits 확률 분포)을 그대로 학생에게 이식하는 과정이다. FP16/BF16에서는 원본의 미세한 정밀도 표현이 유지되지만, INT4/INT8로 양자화(Quantization)되면 확률값들의 미세한 차이가 뭉개지거나(Clipping/Rounding Error) 왜곡된다.

#### ② 추론 능력(Reasoning) 및 에지 케이스 성능 하락 ####
100B급 거대 모델을 양자화하면 일반적인 질문에 대한 답은 비슷하게 내놓는 것처럼 보인다. 하지만 복잡한 논리 추론, 코드 생성, 아주 정교한 도메인 지식, 에지 케이스(Edge Case) 처리 능력에서는 성능 저하가 눈에 띄게 발생한다. 정제 모델이 배워야 할 가장 소중한 '고난도 지식'이 손실되는 셈이다.

#### ③ '쓰레기 데이터 입력 = 쓰레기 모델 생성' (GIGO) ####
정제 모델(Student)의 성능 상한선은 Teacher 모델이 생성한 데이터의 품질이 결정한다. 인프라 비용 몇 백만 원 아끼려다 학생 모델의 성능 품질 자체가 하향평준화되면, 나중에 모델을 다시 학습시켜야 해서 결국 더 큰 리소스와 시간을 낭비하게 된다.


### 1. VRAM 산정 및 양자화(Quantization) 전략 ###
모델의 정밀도(Precision)에 따라 필요한 VRAM 용량이 결정되며, 추론 시 모델 가중치 용량 + KV Cache(문맥 유지 메모리) 오버헤드가 함께 고려되어야 한다.


| 정밀도 (Precision) | 가중치 용량 | 권장 최소 VRAM (KV 캐시 포함) | 추천 활용 목적 |
| :--- | :--- | :--- | :--- |
| **FP16 / BF16 (원본)** | 약 200 GB | 256 GB ~ 320 GB+ | 무손실 최고 성능, 높은 동시 요청 처리 |
| **FP8 / INT8** | 약 100 GB | 128 GB ~ 160 GB | 성능 손실 최소화 및 메모리 효율성 극대화 |
| **INT4 (AWQ / GPTQ)** | 약 50~60 GB | 80 GB ~ 96 GB | 비용 절감 중심 및 단일 노드 가성비 세팅 |


### 2. EC2 GPU 인스턴스 선정 ###

#### 🟢 옵션 A: INT4 양자화 모델 (가성비 세팅) ####
*	추천 인스턴스: g5.12xlarge (NVIDIA A10G 24GB x 4개 = 96GB VRAM)
*	특징: 비용 효율성이 가장 뛰어나고,INT4로 양자화된 100B 모델을 Tensor Parallelism(TP=4)으로 올리기에 최적이다.

#### 🟡 옵션 B: FP8 / INT8 양자화 모델 (밸런스형 세팅) ####
*	추천 인스턴스: g5.48xlarge (NVIDIA A10G 24GB x 8개 = 192GB VRAM) 또는 g6.48xlarge (NVIDIA L4 24GB x 8개 = 192GB VRAM)
*	특징: 8개의 GPU를 묶어(TP=8) INT8 또는 FP8 모델을 여유 있게 로드하고 동시 접속자 처리 폭을 늘릴 수 있다.

#### 🔴 옵션 C: FP16 원본 또는 고성능 대규모 서비스 (최고 성능) ####
*	추천 인스턴스: p4d.24xlarge (NVIDIA A100 40GB x 8개 = 320GB VRAM) 또는 p4de.24xlarge / p5.48xlarge (A100 80GB / H100 80GB x 8개 = 640GB VRAM)
*	특징: 원본 FP16 모델을 그대로 띄우거나, 긴 컨텍스트(Long Context) 및 높은 동시 처리량(Throughput)을 보장해야 할 때 사용한다.


### 3. 추론 프레임워크 선택 (vLLM 권장) ###
100B 모델을 여러 GPU로 분산 처리하려면 Tensor Parallelism(텐서 병렬화)과 PagedAttention을 지원하는 추론 엔진이 필수적이다.
* vLLM (가장 추천): 설치 및 설정이 간편하며, 처리량(Throughput)이 뛰어나고 OpenAI 호환 API 서버를 즉시 제공한다.
* TensorRT-LLM: NVIDIA GPU 커널 최적화를 통해서 레이턴시를 vLLM 에 비해서 줄일수 는 있지만 설정이 복잡하고 별도의 컴파일 타임이 필요하다.


### 4. vLLM 기반 실전 배포 가이드 (예시) ###

```
pip install vllm
```
* 8개 GPU 환경에서 FP16/BF16 모델 실행 (예: TP=8)
```
python3 -m vllm.entrypoints.openai.api_server \
    --model <your-100b-model-path-or-repo> \
    --tensor-parallel-size 8 \
    --max-model-len 4096 \
    --gpu-memory-utilization 0.90 \
    --port 8000
```

* 4개 GPU 환경에서 INT4 AWQ 모델 실행 (예: TP=4)
```
python3 -m vllm.entrypoints.openai.api_server \
    --model <your-100b-awq-model-path> \
    --quantization awq \
    --tensor-parallel-size 4 \
    --max-model-len 4096 \
    --port 8000
```

### 5. 필수 체크 및 최적화 포인트 ###

#### 1.	디스크 용량 확보 (EBS Volume) ####
100B 모델 가중치 다운로드 및 캐시용으로 최소 300GB~500GB 이상의 GP3 EBS 디스크가 필요하다. IOPS 및 대역폭 수치를 올려두면 모델 로딩 시간이 단축된다.

#### 2.	GPU 간 NVLink 대역폭 확인 ####
100B 모델은 GPU 간 통신 병목이 성능에 큰 영향을 준다.. 단일 노드 안에서 AWS 의 G 시리즈 인스턴스는 NVLink 를 제공하지 않으므로 TP 동작시 EFA 를 통한 통신이 발생한다.
분산 인퍼런스 과정에서 레이턴시 관련 이슈가 있는 경우 NVLink 를 제공하는 P 시리즈를 사용해야 한다.

#### 3.	비용 절감 전략 ####
* 서버를 24시간 띄워두지 말고, 데이터 생성 프롬프트 시나리오를 미리 다 만들어 둔 뒤 A100/H100 8장짜리 EC2 인스턴스를 스팟(Spot) 또는 필요할 때만 켜서(On-Demand) 몇 시간 동안 바짝 데이터만 뽑아내고 즉시 인스턴스를 종료(Terminate) 한다.
* 실시간 응답(Latency)은 신경 쓸 필요가 없으므로, vLLM 설정 시 max_num_seqs나 batch_size를 최대한으로 올려 초당 데이터 생성량(Throughput)을 극대화하면 인스턴스 가동 시간을 대폭 줄일 수 있다.


## 멀티 모달 지식 증류 ##

### 1. 멀티모달 증류에서 Teacher 모델이 해주는 역할 ###
멀티모달 모델 정제(Distillation)의 핵심은 "이미지/영상 프레임이라는 시각적 입력을 보고, 얼마나 정교하고 논리적인 시각적 이해(Visual Reasoning)와 묘사를 뽑아내는가" 이다.

이때 Teacher 모델이 수행하는 대표적인 작업은 아래 3가지이다.
#### 1.	고품질 시각-텍스트 데이터셋 생성 (Synthetic Visual Data) ####
입력된 이미지나 영상 프레임에 대해 상세 캡셔닝(Detailed Captioning), 객체 간 관계 설명, 시각적 Q&A(VQA) 데이터를 고품질로 생성한다.
#### 2.	Chain-of-Thought (CoT) 시각적 추론 과정 생성 ####
단순히 "이 영상은 무엇이다"라고 정답만 말하는 것이 아니라, "0초~3초 지점에서 X가 나타나고, Y 행동을 취했으므로 Z다"와 같이 시간/공간적 추론 과정을 텍스트로 풀어내어 학생 모델에 학습시킨다.
#### 3.	Vision-Text Cross-Attention 또는 Logit 이식 (선택적) ####
모델 구조가 호환된다면 시각 토큰(Visual Token)과 텍스트 토큰 간의 Attention Map이나 Logit 분포를 이식한다.

### 2. 영상(Video) 처리 시 VRAM 관리 핵심 ###
100B 멀티모달 모델에 영상 데이터를 그대로 넣으면 시각 토큰(Visual Tokens) 수가 폭증하여 VRAM 경고가 뜨기 쉽다.
* 프레임 샘플링(Frame Sampling): 영상 전체를 넣기보다 1초당 1~2프레임 또는 주요 키프레임(Keyframe)만 추출하여 텐서로 변환 후 Teacher에 입력한다.
* Image Resolution 조정: 입력 이미지/프레임의 해상도를 적절히 자르거나(Crop/Resize) 픽셀 수를 제한하여 인퍼런스 속도(Throughput)를 확보한다.
  
### 3. 멀티모달 Teacher 인퍼런스 세팅 체크리스트 ###
* 추론 프레임워크 - 최근 vLLM과 SGLang 모두 LLaVA, Qwen-VL, InternVL 등 멀티모달 모델의 Multi-Image/Video 추론과 Tensor Parallelism을 강력하게 지원
* 정밀도 (Precision) - FP16 / BF16 고정, 시각 임베딩 레이어와 언어 모델 레이어 간의 정밀도 차이로 인해 양자화 시 비전 인지 능력이 크게 떨어질 수 있으므로 원본 정밀도 유지.
* GPU - p4d.24xlarge (A100 40GB x 8) 또는 p4de / p5, 시각 토큰으로 인해 KV Cache 메모리 소모가 일반 텍스트보다 훨씬 크므로 GPU VRAM 여유가 필수적.

데이터 생성 파이프라인이 제대로 동작하는지, 1B 모델이 OOM 없이 도는지 1장 세팅으로 먼저 몇 백 스텝 돌린다.

#### 2.	본 학습 및 정제 모델 훈련 (2장 사용 - 베스트) ####
* 검증이 끝나면 2장 구성 + PyTorch Native DDP로 본격적인 학습을 진행한다.
* 48GB x 2장 환경이라면 DeepSpeed 복잡한 설정 없이 torchrun 기반 DDP만으로도 최상의 속도와 안정성을 확보할 수 있다.

#### 3.	데이터가 너무 커서 2장으로도 며칠씩 걸릴 때 (4장으로 확장): ####

```
torchrun \
  --nnodes=2 \
  --nproc-per-node=4 \
  --max-restarts=3 \
  --rdzv-id=job_123 \
  --rdzv-backend=c10d \
  --rdzv-endpoint=master_node_ip:29500 \

torchrun \
  --nproc-per-node=4 \
  train.py --batch_size 2

torchrun --standalone --nproc-per-node=4 train.py
```
* NCCL_P2P_DISABLE=1을 설정하여 CPU System RAM 우회 모드로 돌리는 것이 인스턴스 내부 통신 대역폭을 가장 알차게 끌어쓰는 가장 빠르고 유일한 정석(?)


## G6e 통신 아키텍처 ##
### 1. 물리적 대역폭(Bandwidth)의 압도적 차이 ###
* System Memory (CPU RAM) 경유 통신:
  - GPU0 ➔ PCIe 버스 ➔ CPU Host RAM (DDR5) ➔ PCIe 버스 ➔ GPU1
  - g6e 메인보드 및 최신 CPU 버스 대역폭은 초당 최소 100GB/s ~ 200GB/s 이상을 처리.

* EFA (Elastic Fabric Adapter) 루프백 통신:
  -	GPU0 ➔ PCIe ➔ CPU ➔ EFA NIC (네트워크 카드) ➔ (외부 스위치/루프백) ➔ EFA NIC ➔ CPU ➔ PCIe ➔ GPU1
  -	g6e 인스턴스 전체에 붙어있는 EFA의 최대 대역폭은 다 끌어모아봤자 400Gbps = 초당 50GB/s 수준.

결론: 메모리를 통한 내부 복사는 최소 100~200GB/s, EFA는 최대 50GB/s로 System Memory 경로가 최소 2~4배 이상 넓은 통신 도로를 제공.

### 2. 지연 시간(Latency) 및 Protocol Overhead 차이 ###
* System Memory 방식:  
  PyTorch/NCCL의 SHM (Shared Memory) 모드로 동작하여, 순수하게 DRAM 메모리 영역에 주소를 지정하고 C++ 단에서 빠르게 데이터를 복사(memcpy).
* EFA 방식:
  굳이 안 나가도 되는 데이터를 외부 네트워크 카드까지 내려보내기 위해 AWS SRD 프로토콜 패킷으로 캡슐화(Encapsulation)하고, 네트워크 링버퍼를 통과시킨 뒤 다시 패킷을 풀어내는 무거운 오버헤드가 추가.

### 3. 결정적으로: g6e의 EFA는 GPUDirect RDMA를 지원하지 않음 ###
P4d/P5 같은 최고급 인스턴스는 GPU가 EFA 카드로 직접 데이터를 쏘는 GPUDirect RDMA가 하드웨어로 뚫려 있다.
그러나 g6e(L40S) 인스턴스는 GPUDirect RDMA를 지원하지 않는다.
따라서 g6e에서 EFA를 써서 통신하려고 해도, 어차피 데이터를 EFA 카드로 전달하기 위해 CPU Host Memory를 한 번 거쳐야만 한다.
* System Memory 방식: GPU ➔ CPU 메모리 ➔ GPU (2단계)
* EFA 방식: GPU ➔ CPU 메모리 ➔ EFA 카드 ➔ EFA 카드 ➔ CPU 메모리 ➔ GPU (4단계 + 네트워크 패킷 처리)
